git常用命令

发表于 2017-10-12 | 分类于 git | 阅读次数:

记录一些常用的不同使用场景的 git 命令。

参考：

  • Pro Git book
  • 廖雪峰的官方网站

基本概念

  • Working Directory：工作区，从仓库某个版本提取出来的内容，可以简单理解成存放本地仓库的文件夹
  • Staging Area：暂存区，存放下次提交的文件列表信息
  • Git Directory（Repository）：仓库，存放项目元数据与对象数据库的地方，克隆仓库时，便是拷贝这里的数据

git 基本的工作流程是：

  1. 在 Working Directory 中修改
  2. 将修改的文件快照暂存到 Staging Area
  3. 提交更新，将 Staging Area 的文件，存到 Git Directory，生成一条提交历史

往仓库添加文件、从仓库删除文件（Tracked files & Untracked files）

新加文件后，文件并没有加入 Staging Area，需要使用 git add 命令，给文件添加索引，才会加入 Staging Area。
没有暂存到 Staging Area 的文件，是不能被追踪交提交到 Git Directory 的。

git add path        # 追踪path目录下所有文件（git add . 就是追踪当前目录下所有文件）
git add filename    # 追踪当前目录下的filename文件

git rm -f filename    # 删除对filename的追踪并将filename文件从目录中删除
git rm --cached filename    # 删除对filename的追踪，filename文件保留在本地
git rm -r -f dir    # 删除对dir目录的追踪并将dir目录从当前目录中删除
git rm -r --cached dir    # 删除对dir目录的追踪，dir目录保留在本地

git statu 可以查看 Staging Area 的当前状态，看有哪些修改内容。

要被 git 忽略的文件，直接在 .gitignore 中添加就可以了，之后用 git status 就不会被列出

提交修改

Staging Area 中暂存的文件，使用 git commit 命令才真正提交到 Git Directory。

git commit -m "commit log"    # commit log是此次提交的log，一般标注此次修改的内容

还原 Working Directory 内的修改

修改了 Working Directory 中的文件后，但未 git add 到 Staging Area。如果想恢复 Working Directory 中已修改的文件到没修改前的样子，可以使用命令：

git checkout -- .            # 全部还原
git checkout -- filename        # 还原单个文件 filename

还原 Staging Area 内的修改

如果 Working Directory 中修改的内容被 git add 到 Staging Area，想恢复可以用命令：

git reset HEAD                # 全部还原
git reset HEAD filename        # 还原单个文件 filename

这时 Staging Area 的修改被还原到了 Working Directory，再用前面还原 Working Directory 的指令就可以恢复到未修改之前的样子。HEAD 表示恢复到最近提交一个版本。

这里没指定 reset 模式，则默认是 --mixed，上面命令等价于 git reset --mixed HEAD filename，表示重置 HEAD 到 HEAD commit(这里表示 HEAD 保持不变），重置 index，working copy 不会更改。所有 HEAD commit 后的修改，会回到 working copy 中，不会存在于 Staging Area，可以再重新 git add 到 Staging Area。

还原 Git Directory 到之前的版本

已经提交了不合适的修改到 Git Directory 时，想要撤销本次提交，可以使用版本回退指令，不过前提是没有推送到远程库。

先用 git log 查看要回退的版本号：

$ git log
commit d3e59e7a450387a1694d142b80416bf0ba593a63
Author: xxxxxx
Date:   Thu Oct 12 11:07:00 2017 +0800

    new branch1

commit 20120028f04f9338e5c70536ea6a815618bd3f1c
Author: xxxxxx
Date:   Thu Oct 12 11:07:41 2017 +0800

    new master

假设回退到版本 20120028f04f9338e5c70536ea6a815618bd3f1c：

$ git reset --hard 20120028f04f9338e5c70536ea6a815618bd3f1c
HEAD is now at 2012002 branch1 modify a.txt

此时就回退到 20120028f04f9338e5c70536ea6a815618bd3f1c 版本了，其中 --hard 参数表示将重置 HEAD 返回到另外一个 commit，并抹去所有版本 20120028f04f9338e5c70536ea6a815618bd3f1c 后修改的内容（git log 将不再看到 20120028f04f9338e5c70536ea6a815618bd3f1c 后的版本内容），包括原先 Staging Area 中待提交的内容。

还有其他参数（--soft、--mixed），这是区别

合并 merge

git 中合并不同分支使用 git merge 命令：

git merge branch

把 branch 分支的内容合并到当前分支。切换当前分支使用 git checkout 命令：

git checkout brach

将当前分支改为 branch 分支。

合并后可能会有冲突，编辑冲突文件后，重新 git add 和 git commit 完成分支的合并。

变基 rebase

  • 参考1
  • 参考2

git 中合并不同分支另一种命令，get rebase，与 git merge 的不同是，get rebase 合并后，主分支的提交历史会简洁一些，一条时间线，而 git merge 合并后，主分支的提交历史中会并入从分支中的提交历史。

假设现在有 master、branch 两个分支，各自有修改，现要把 branch 合入 master，以 rebase 方式合并的流程是：

git checkout branch
git rebase master
git checkout master
git merge branch

因为 master 与 branch 各自有修改，所以 git rebase master 时可能会有冲突。

rebase 冲突解决过程，就是解决每个应用补丁冲突的过程。解决完一个补丁应用的冲突后，执行下面命令标记冲突已解决（也就是把修改内容加入缓存）：

git add -u

-u 表示把所有已 track 的文件的新的修改加入缓存，但不加入新的文件。

然后执行下面命令继续 rebase：

git rebase --continue

如果还有冲突，则继续解决，重复这些步骤，直到 rebase 完成。

rebase 之后，不需要执行 commit，也不存在新的修改需要提交，都是git自动完成。

如果中间遇到某个补丁不需要应用，可以用下面命令忽略：

git rebase --skip

如果想回到rebase执行之前的状态，可以执行：

git rebase --abort

分支操作

查看仓库内分支

git branch        # 显示所有分支，并高亮显示当前的分支

创建分支命令：

git branch newbranch    # 创建新分支但不自动切换到新分支，名为 newbranch，新分支的内容与当前分支一样

创建分支并并自动切换到新分支命令：

git checkout -b newbranch    # 创建并自动切换到新分支 newbranch，新分支的内容与当前分支一样

切换到其他分支：

git checkout otherbranch    # 切换到 otherbranch 分支，如果该分支不存在，切换失败
